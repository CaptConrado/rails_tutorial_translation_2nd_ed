Prefacio

Mi antigua empresa (CD Baby) fue una de las primeras en cambiar fuertemente a Ruby on Rails, y aún con más fuerza volver a PHP (busquame en Google para leer sobre el mi drama). Este libro de Michael Hartl era tan recomendado que tenía que probarlo, y el Tutorial de Ruby on Rails es lo que yo necesidava para volver a Rails otra vez.

Aunque he trabajado a mi manera a través de muchos libros de Rails, este libro finalmente me hizo "absorber" Rails. Todo se hace mucho en “the Rails way” — Un camino “a way” que se sentía muy natural para mí, pero ahora después de leer este libro finalmente se siente natural. Este es también el único libro de Rails that does test-driven development the entire time, an approach highly recommended by the experts but which has never been so clearly demonstrated before. Por último, mediante la inclusión de Git, GitHub, y Heroku en los ejemplos de demostración, el autor realmente te da una idea de lo que es hacer un proyecto en el mundo real. Los ejemplos de código fuente de este tutorial no están en aislamiento.

La narración que usa esto libro es una solución genial. Personalmente, me puse a estudiar a fondo el Tutorial Rails en tres largos días, haciendo todos los ejemplos y desafíos al final de cada capítulo. Hágalo desde el principio hasta el final, sin saltos, y obtendrá el máximo beneficio.

Capitulo 1 Desde cero al primero código

Bienvenidos al Tutorial de Ruby on Rails . El objetivo de este libro es la mejor respuesta a la pregunta, “Si quiero aprender el desarrollo web con Ruby on Rails, de donde debería de empezar?” En el momento en que termine el Tutorial de Ruby on Rails, Usted tendrá todas las habilidades que necesitan para desarrollar e implementar sus propias aplicaciones Web personalizadas con Rails. También estará listo para beneficiarse de los muchos libros más avanzados, blogs, y videos que forman parte de la floreciente ecosistema educativo Rieles. Finalmente, desde cuando el Tutorial de Ruby on Rails usa Rails 3, los conocimientos que adquiera aquí representa el estado del arte en el desarrollo web. (La versione más actualizada del Tutorial de Ruby on Rails se puede encontrar en el sitio de libro a http://railstutorial.org/; si Usted está leyendo este libro fuera línea, asegúrate de revisar la version en línea del Tutorial de Rails a http://railstutorial.org/book para las actualizaciones más recientes.)

Tenga en cuenta que el objetivo de este libro no es el simple hecho de enseñar Rails, sino más bien para enseñar para el desarrollo web con Rails, lo que significa la adquisición de (o ampliar) las habilidades necesarias para desarrollar software para la World Wide Web. Además de Ruby on Rails, este conjunto de habilidades incluye HTML & CSS, base de datos, control de versiones, pruebas y deployment. Para lograr este objetivo, el Tutorial de Ruby on Rails adopta un enfoque integrado: usted aprenderá Rails por ejemplos mediante la creación de una aplicación de ejemplo cero. Como Derek Sivers notas en el prólogo, este libro está estructurado como un narrativa lineal, diseñado para ser leído de principio a fin. Si estás acostumbrado a saltar pàginas en libros técnicos, este enfoque lineal podría requerir algunos ajustes, pero te sugiero que darle una oportunidad. Tu puede pensar que el Tutorial de Ruby on Rails es como un video juego donde tú eres el protagonista y tu subes de nivel como desarrollador Rails en cada capítulo. (Los ejercicio son como minibosses.)

En el primero capitulo, vamos a empezar con Ruby on Rails mediante la instalación de todo el software necesario y mediante la creación de nuestro entorno de desarrollo (Seccion 1.2). A continuación, vamos a crear nuestra primera aplicación Rails, llamada (muy apropiadamente) first_app(primera applicacion). El Tutorial de Rails hace hincapié en las buenas prácticas de desarrollo de software, por lo que inmediatamente después de la creación de nuestro nuevo proyecto de Rails lo vamos a poner bajo control de versiones con Git (Seccion 1.3). Y, créanlo o no, en este capítulo vamos a poner la nuestra primera aplicación en la red más amplia mediante la implementación de una distribucion en produccion (Seccion 1.4).

In Capitulo 2, vamos a hacer un segundo proyecto, cuyo propósito es demostrar el funcionamiento básico de una aplicación Rails. Para ponernos en marcha rápidamente, vamos a construir este aplicación de demostración (llamada demo_app) usando el scaffolding (Box 1.1) para generar el código, ya que este código es complejo y feo, Capitulo 2 se centrará en la interacción con la aplicación de demostración a través de su URIs (a veces llamado URLs)1 usando un navegador.

El resto del tutorial se centra en el desarrollo de un único gran ejemplo de aplicación (llamado sample_app), escribiendo todo el código desde cero. Vamos a desarrollar la sample_app usando test-driven development (TDD), para empiezar en el Capitulo 3 mediante la creación de páginas estáticas y luego agregar un poco de contenido dinámico. Vamos a dar un rápido desvío en el Capitulo 4 para aprender un poco sobre el lenguaje Ruby subyacente Rails. Entonces, in Capitulo 5 a través de Capitulo 9, vamos a completar las bases para la aplicación de ejemplo al hacer un diseño de sitio, un modelo de datos de usuario, y un sistema completo de registro y autenticación. Finally, en el Chapter 10 and Capitulo 11 vamos a añadir funciones de microblogging y de reds sociales para hacer un sitio de ejemplo que funciona.

la aplicación final tendrá una cierta similitud con la más común y popolares sitios de microblogging sociales—un sitio que, coincidentemente, también fue escrito originalmente en Rails. A pesar de la necesidad que nuestros esfuerzos se centran en la aplicación de ejemplo específico, el énfasis en todo el Tutorial de Rails estará a los principios generales, de modo que usted tendrá una base sólida no importa qué tipo de aplicaciones web que desea construir.

Box 1.1.Scaffolding: Más rápido, más fácil, más seductor

Desde el principio, Rails se ha beneficiado de una palpable sensación de emoción, empezando por el famoso 15-minute weblog video de el creador de Rails David Heinemeier Hansson. Ese video y sus sucesores son una gran manera de obtener una muestra del poder de Rails’, y recomiendo de verlos. Pero cuidado: cumplan su increíble hazaña de quince minutos usando una característica llamada scaffolding, que se basa en gran medida en generar código fuente, mágicamente creado gracias al comado generacion de Rails.

Al escribir un tutorial de Ruby on Rails, existe la tentación de confiar en el enfoque de scaffolding—Es más rápido, más fácil, más seductor. Sin embargo, la complejidad y la enorme cantidad de código en el scaffolding puede ser completamente abrumador para el comienzo de un desarrollador en Rails; usted puede ser capaz de usarlo, pero es probable que no lo entiende. Siguiente el enfoque de scaffolding existe el riesgo de convertirte en un virtuoso generador de scripts con poco (y frágil) conocimiento real de Rails.

En el Tutorial de Ruby on Rails, vamos a tomar el (casi) enfoque opuesto: aunque en elCapitulo 2 se desarrollará una aplicación de demostración pequeño usando scaffolding, el corazon de el Tutorial de Rails es la sample app, que vamos a empezar a escribir en Capitulo 3. En cada etapa de desarrollo de la aplicación de ejemplo, vamos a escribir pequeño y tamaño de un bocado pedazo de codigo—bastante simple de entender, pero la novela sèra como un desafìo. El efecto acumulativo será una más profunda y más flexible conocimiento de Rails, que le da una buena base para escribir casi cualquier tipo de aplicación web.

1.1 Introducción

Desde su debut en 2004, Ruby on Rails se ha convertido rápidamente en uno de los marcos más potentes y populares para crear aplicaciones web dinámicas. Todos, desde startups a grandes empresas han utilizado Rails: 37signals, GitHub, Shopify, Scribd, Twitter, LivingSocial, Groupon, Hulu, Yellow Pages—y la lista de sitios que utilizan podría continuar. También hay muchas tiendas de desarrollo web especializadas en Rails, como ENTP, thoughtbot, Pivotal Labs, and Hashrocket, además innumerables consultores independientes, entrenadores y contratistas.

Lo que hace que los carriles tan grande? En primer lugar, Ruby es 100% de código abierto, disponible bajo la MIT License, también no cuesta nada para descargar o utilizar. Rails también le debe mucho de su éxito a su diseño elegante y compacto; aprovechando la maleabilidad del subyacente lenguaje de Ruby, Rails efectivamente crea un Lenguaje específico del dominio para escribir aplicaciones web. Como resultado de ello, muchas tareas comunes de programación web, tales como la generación de código HTML, por lo que los modelos de datos y enrutamiento URI-son fáciles con Rails, y el código de la aplicación resultante es concisa y fácil de leer.

Rails también se adapta rápidamente a los nuevos desarrollos en la tecnología web y framework. Por ejemplo, Rails fue uno de los primeros marcos para digerir completamente y aplicar el estilo arquitectónico REST para la estructuración de las aplicaciones web (que vamos a aprender acerca de todo este tutorial). Y cuando salieron otros framework de desarrollar con nuevas técnicas, el creator de Rails David Heinemeier Hansson y el corazon del equipo de rails no dudaron en incorporar sus ideas. Tal vez el ejemplo más dramático es la fusión de Rails y Merb, un framework web rival en Ruby, por lo que Rails ahora se beneficia de diseño modular de Merb, estable API, y un rendimiento mejorado.

Por último, Rails se beneficia de una comunidad entusiasta y extraordinaria. Los resultados incluyen centenares de código abierto contribuyentes, buena asistencia conferencias, un gran número de plugins y gemas (que dan autónomos soluciones a problemas específicos como la paginación y la carga de imágenes), una rica variedad de blogs informativos, y una gran cantidad de foros de discusión y los canales de IRC. El gran número de programadores de Rails también hace que sea más fácil manejar los errores inevitables de la aplicación: el “Google the error message” algoritmo casi siempre produce una lista de blog que sea pertinente o foros con muchos comentarios.

1.1.2 “Scaling” Rails

Antes de continuar con el resto de la introducción, me gustaría tomar un momento para abordar sobre el problema que està desde los inicios de el nacimiento del framework de Rails: la supuesta incapacidad de Rails a "escalar", es decir, a manejar grandes cantidades de tráfico. Parte de este problema se basó en una interpretación errónea; tu puede escalar un sitio, no un framework, y Rails es solo un framework. Así que la verdadera pregunta debería haber sido, "¿Puede un sitio construido con Rails ser escaldo?" En cualquier caso, la cuestión ha sido definitivamente una respuesta afirmativa: algunos de los sitios de mayor tráfico del mundo utilizan Rails. En realidad hacer la escala es más allá del alcance de sólo Rails, pero està seguro que la tuya aplicación nunca necesiterà el cargo de Hulu o las Yellow Pages, Rails no le impedirá tomar el mundo.

1.1.3 Convenciones de este libro

Las convenciones que aparecen en este libro son en su mayoría auto-explicativo. En esta sección, voy a mencionar algunas que no lo son.

Tanto el HTML y PDF ediciones de este libro están llenas de enlaces, tanto a las secciones internas (tal como Seccion 1.2) y a los sitios externos (como la principal pàginas para descargar Ruby on Rails ).5

Hay muchos ejemplos en este libro en el uso de línea de comandos. Por razones de simplicidad, todos los ejemplos de línea de comandos de Unix utiliza un estilo de línea de comandos (un signo de dólar), de la siguiente:

$ echo "hello, world"
hello, world

Los usuarios de Windows deben entender que sus sistemas se utiliza el indicador de ángulo análogo >:

C:\Sites> echo "hello, world"
hello, world

En los sistemas Unix, algunos comandos deben ser ejecutados con sudo, que reemplaza los usuario con lo administrador del sistema;.6 De forma predeterminada, el comando ejecutado con sudo se ejecuta como un usuario administrativo, que tiene acceso a los archivos y directorios que los usuarios normales no pueden tocar, como en este ejemplo de Secciòn 1.2.2:

$ sudo ruby setup.rb

La mayoría de los sistemas Unix/Linux/OS X requieren sudo por defecto, a menos que usted está utilizando de Ruby Version Manager como se sugiere en Sección 1.2.2.3; en este caso, tendría que escribir esto en su lugar:

$ ruby setup.rb

Rails viene con un montón de comandos que se pueden ejecutar en la línea de comandos. Por ejemplo, en Secciòn 1.2.5 vamos a correr un servidor web para desarrollo local con el comando siguiente:

$ rails server

As with the command-line prompt, the Tutorial de Rails uses the Unix convention for directory separators (ejemplo, una diagonal /). La mia aplicacion de ejemplo para el Tutorial de Rails, por ejemplo, vive en

/Users/mhartl/rails_projects/sample_app

En Windows, el directorio análogo sería

C:\Sites\sample_app

The root directory for any given app is known as the Rails root, pero esta terminología es muy confundido y muchas personas creen erróneamente que la "Rails root"; es el directorio raíz para Rails en sí. Para mayor claridad, el Tutorial de Rails se referirá a la raíz Rails como la raíz de la aplicación , y en lo sucesivo todos los directorios serán relativos a este directorio. Por ejemplo, el config de mi aplicación de ejemplo es 

/Users/mhartl/rails_projects/sample_app/config

El directorio raíz de la aplicación aquí es todo antes config, ejemplo,

/Users/mhartl/rails_projects/sample_app

Por razones de brevedad, se refiere a la archivo

/Users/mhartl/rails_projects/sample_app/config/routes.rb

Voy a omitir la raíz de la aplicación y escribir simplemente config/routes.rb.

El Tutorial de Rails a menudo se muestra el resultado de los diversos programas (comandos de la shell, el estado de control de versiones, programas de Ruby, etc.). Debido a las innumerables diferencias pequeñas entre los diferentes sistemas informáticos, la salida (output) que ves no siempre están de acuerdo exactamente con lo que aparece en el texto, pero esto no es motivo de preocupación.

Algunos comandos pueden producir errores en función de su sistema, en vez de intentar el Sisyphean que tiene la tarea de documentar todos los errores de este tipo en este tutorial, voy a delegar en el algoritmo de "Busqua en Google el mensaje de error", que entre otras cosas es una buena práctica para el desarrollo de software en la vida real. Si surge algún problema al seguir el tutorial, le sugiero consultar a los recursos que se enumeran en el Página de ayuda del Tutorial de Rails.7

Como parte de la instalación de Ruby, si usted está usando OS X o Linux recomiendo usarRuby Version Manager (RVM), que le permite instalar y gestionar múltiples versiones de Ruby en la misma máquina. (E lproyecto Pik logra una hazaña similar en Windows.) Esto es particularmente importante si se desea ejecutar diferentes versiones de Ruby o Rails en la misma máquina. Si surge algún problema con RVM, usted puede encontrar a menudo a su creador, Wayne E. $ rvm get head && rvm reload
$ rvm install 1.9.3
<wait a while>

Aquí el primero comando es actualizar y vuelver a cargar RVM, que es una buena práctica, ya RVM se actualiza con frecuencia. El segundo instala la versión 1.9.3 de Ruby, dependiendo de su sistema, que podría tomar un tiempo para descargar y compilar, así que no te preocupes si parece estar tomando para siempre.

$ brew install automake
$ rvm install 1.9.3

$ rvm install 1.9.3 --with-openssl-dir=$HOME/.rvm/

En algunos sistemas antiguos OS X, es posible que tenga que incluir la ruta a la library llamada readline:

$ rvm install 1.9.3 --with-readline-dir=/opt/local

(Como he dicho, muchas cosas pueden salir mal. La única solución es la búsqueda en la web y la determinación.)

A los efectos de este tutorial, te sugiero la creación de un llamado gemset rails3tutorial2ndEd:

$ rvm use 1.9.3@rails3tutorial2ndEd --create --default
Using /Users/mhartl/.rvm/gems/ruby-1.9.3 with gemset rails3tutorial2ndEd

Este comando crea (--create) el gemset rails3tutorial2ndEd asociado con Ruby 1.9.3 mientras que la organización para empezar a usarlo inmediatamente (use) y establecer como predeterminado (--default) gemset, por lo que cada vez que abrimos una nueva ventana de terminal del 1.9.3@rails3tutorial2ndEd la combinación Ruby/gemset se selecciona automáticamente. RVM soporta una gran variedad de comandos para manipular gemsets; consulte la documentación en http://rvm.beginrescueend.com/gemsets/. Si alguna vez te quedas atascado con RVM, ejecutar comandos como estos deben ayudarle a conseguir la solucion:

$ rvm --help
$ rvm gemset --help

Instalación de RubyGems

RubyGems es un gestor de paquetes para los proyectos de Ruby, y hay muchas librerías (library) útiles (incluyendo Rails) disponibles como paquetes de Ruby, o gemas (gems). Instalar RubyGems debe ser fácil una vez que instale Ruby. De hecho, si usted tiene instalado RVM, ya tiene RubyGems, porque ya incluye automáticamente RVM:

$ which gem
/Users/mhartl/.rvm/rubies/ruby-1.9.3-p0/bin/gem

Si no lo tiene, debe descargar RubyGems, extraerlo, y luego ir al directorio de rubygemsy ejecutar la instalación de programa:

$ ruby setup.rb

(Si obtiene un error de permisos aquí, ir a la Sección 1.1.3 puede que tenga que utilizar sudo.)

Si ya han RubyGems instalado, debe asegurarse que su sistema utiliza la versión utilizada en este tutorial:

$ gem update --system 1.8.24

Congelar el sistema para esta versión en particular le ayudará a prevenir conflictos como los cambios de RubyGems en el futuro.

Al instalar las gemas (gems), de forma predeterminada RubyGems genera dos tipos de documentación (llamados ri y rdoc), pero muchos desarrolladores de Ruby on Rails piensan que no vale la pena esperar la copia en local de esa documentación. (Muchos desarrolladores se basan en la documentación en línea en lugar de la documentación ri y rdoc nativas.) Para evitar la generación automática de la documentación, yo recomendaría hacer un archivo de configuración llamado .gemrc en el directorio principal Listing 1.1 con la linea en Listing 1.2. (La tilde "~" significa "directorio principal", mientras que el punto en.gemrc hace que el archivo es oculto, es una convención común para los archivos de configuración. )

Listing 1.1. Creación de un archivo de configuración para las gemas 
$ subl ~/.gemrc

Aquí subl es el comando de línea de comandos para lanzar Sublime Text en OS X, que se puede configurar mediante la Sublime Text 2 documentation for the OS X command line. Si estás en una plataforma diferente, o si usted está usando un editor diferente, deberá cambiar este comando (ejemplo, haciendo doble clic en el icono de la aplicación o usando un comando alternativo, como mate, vim, gvim, o mvim). Por razones de brevedad, en todo el resto de este tutorial voy a utilizar subl como una abreviatura de "abrir con tu editor de texto favorito."

Listing 1.2. La supresión de la documentación ri y rdoc.gemrc. 
install: --no-rdoc --no-ri
update: --no-rdoc --no-ri

1.2.6 Model-view-controller (MVC)

Even at this early stage, it’s helpful to get a high-level overview of how Rails applications work (Figure 1.5). You might have noticed that the standard Rails application structure (Figure 1.2) has an application directory called app/ with three subdirectories: models, views, and controllers. This is a hint that Rails follows the model-view-controller (MVC) architectural pattern, which enforces a separation between “domain logic” (also called “business logic”) from the input and presentation logic associated with a graphical user interface (GUI). In the case of web applications, the “domain logic” typically consists of data models for things like users, articles, and products, and the GUI is just a web page in a web browser.

When interacting with a Rails application, a browser sends a request, which is received by a web server and passed on to a Rails controller, which is in charge of what to do next. In some cases, the controller will immediately render a view, which is a template that gets converted to HTML and sent back to the browser. More commonly for dynamic sites, the controller interacts with a model, which is a Ruby object that represents an element of the site (such as a user) and is in charge of communicating with the database. After invoking the model, the controller then renders the view and returns the complete web page to the browser as HTML.

Figure 1.5: A schematic representation of the model-view-controller (MVC) architecture.

If this discussion seems a bit abstract right now, worry not; we’ll refer back to this Sección frequently. In addition, Sección 2.2.2 has a more detailed discussion of MVC in the context of the demo app. Finally, the sample app will use all aspects of MVC; we’ll cover controllers and views starting in Sección 3.1.2, models starting in Sección 6.1, and we’ll see all three working together in Sección 7.1.2.

1.3 Version control with Git

Now that we have a fresh and working Rails application, we’ll take a moment for a step that, while technically optional, would be viewed by many Rails developers as practically essential, namely, placing our application source code under version control. Version control systems allow us to track changes to our project’s code, collaborate more easily, and roll back any inadvertent errors (such as accidentally deleting files). Knowing how to use a version control system is a required skill for every software developer.

There are many options for version control, but the Rails community has largely standardized on Git, a distributed version control system originally developed by Linus Torvalds to host the Linux kernel. Git is a large subject, and we’ll only be scratching the surface in this book, but there are many good free resources online; I especially recommend Pro Git by Scott Chacon (Apress, 2009). Putting your source code under version control with Git is strongly recommended, not only because it’s nearly a universal practice in the Rails world, but also because it will allow you to share your code more easily (Sección 1.3.4) and deploy your application right here in the first chapter (Sección 1.4).

1.3.1 Installation and setup

The first step is to install Git if you haven’t yet followed the steps in Sección 1.2.2.2. (As noted in that Sección, this involves following the instructions in the Installing Git Sección of Pro Git.)

First-time system setup

After installing Git, you should perform a set of one-time setup steps. These are system setups, meaning you only have to do them once per computer:

$ git config --global user.name "Your Name"
$ git config --global user.email your.email@example.com

I also like to use co in place of the more verbose checkout command, which we can arrange as follows:

$ git config --global alias.co checkout

This tutorial will usually use the full checkout command, which works for systems that don’t have co configured, but in real life I nearly always use git co.

As a final setup step, you can optionally set the editor Git will use for commit messages. If you use a graphical editor such as Sublime Text, TextMate, gVim, or MacVim, you need to use a flag to make sure that the editor stays attached to the shell instead of detaching immediately:18

$ git config --global core.editor "subl -w"

Replace "subl -w" with "mate -w" for TextMate, "gvim -f" for gVim, or "mvim -f" for MacVim.

First-time repository setup

Now we come to some steps that are necessary each time you create a new repository. First navigate to the root directory of the first app and initialize a new repository:

$ git init
Initialized empty Git repository in /Users/mhartl/rails_projects/first_app/.git/

The next step is to add the project files to the repository. There’s a minor complication, though: by default Git tracks the changes of all the files, but there are some files we don’t want to track. For example, Rails creates log files to record the behavior of the application; these files change frequently, and we don’t want our version control system to have to update them constantly. Git has a simple mechanism to ignore such files: simply include a file called .gitignore in the application root directory with some rules telling Git which files to ignore.19

Looking again at Table 1.1, we see that the rails command creates a default .gitignore file in the application root directory, as shown in Listing 1.6.

1.3.2 Adding and committing

Finally, we’ll add the files in your new Rails project to Git and then commit the results. You can add all the files (apart from those that match the ignore patterns in .gitignore) as follows:

$ git add .

Here the dot ‘.’ represents the current directory, and Git is smart enough to add the files recursively, so it automatically includes all the subdirectories. This command adds the project files to a staging area, which contains pending changes to your project; you can see which files are in the staging area using the status command:20

$ git status
# On branch master
#
# Initial commit
#
# Changes to be committed:
#   (use "git rm --cached <file>..." to unstage)
#
#       new file:   README.rdoc
#       new file:   Rakefile
.
.
.

(The results are long, so I’ve used vertical dots to indicate omitted output.)

$ git commit -m "Initial commit"
[master (root-commit) df0a62f] Initial commit
42 files changed, 8461 insertions(+), 0 deletions(-)
create mode 100644 README.rdoc
create mode 100644 Rakefile

The -m flag lets you add a message for the commit; if you omit -m, Git will open the editor you set in Sección 1.3.1 and have you enter the message there.

It is important to note that Git commits are local, recorded only on the machine on which the commits occur. This is in contrast to the popular open-source version control system called Subversion, in which a commit necessarily makes changes on a remote repository. Git divides a Subversion-style commit into its two logical pieces: a local recording of the changes (git commit) and a push of the changes up to a remote repository (git push). We’ll see an example of the push step in Sección 1.3.5.

1.3.3 What good does Git do you?

It’s probably not entirely clear at this point why putting your source under version control does you any good, so let me give just one example. (We’ll see many others in the chapters ahead.) Suppose you’ve made some accidental changes, such as (D’oh!) deleting the critical app/controllers/ directory:

$ ls app/controllers/
application_controller.rb
$ rm -rf app/controllers/
$ ls app/controllers/
ls: app/controllers/: No such file or directory

Here we’re using the Unix ls command to list the contents of the app/controllers/ directory and the rm command to remove it. The -rf flag means “recursive force”, which recursively removes all files, directories, subdirectories, and so on, without asking for explicit confirmation of each deletion.

Let’s check the status to see what’s up:

$ git status
# On branch master
# Changed but not updated:
#   (use "git add/rm <file>..." to update what will be committed)
#   (use "git checkout -- <file>..." to discard changes in working directory)
#
#       deleted:    app/controllers/application_controller.rb
#
no changes added to commit (use "git add" and/or "git commit -a")

We see here that a file has been deleted, but the changes are only on the “working tree”; they haven’t been committed yet. This means we can still undo the changes easily by having Git check out the previous commit with the checkout command (and a -f flag to force overwriting the current changes):

$ git checkout -f
$ git status
# On branch master
nothing to commit (working directory clean)
$ ls app/controllers/
application_controller.rb

The missing directory and file are back. That’s a relief!

Putting a copy of your Git repository at GitHub serves two purposes: it’s a full backup of your code (including the full history of commits), and it makes any future collaboration much easier. This step is optional, but being a GitHub member will open the door to participating in a wide variety of open-source projects.

1.3.5 Branch, edit, commit, merge

If you’ve followed the steps in Sección 1.3.4, you might notice that GitHub automatically shows the contents of the README file on the main repository page. In our case, since the project is a Rails application generated using the rails command, the README file is the one that comes with Rails (Figure 1.8). Because of the .rdoc extension on the file, GitHub ensures that it is formatted nicely, but the contents aren’t helpful at all, so in this Sección we’ll make our first edit by changing the README to describe our project rather than the Rails framework itself. In the process, we’ll see a first example of the branch, edit, commit, merge workflow that I recommend using with Git.

Figure 1.8: The initial (rather useless) README file for our project at GitHub. (full size)

Branch

Git is incredibly good at making branches, which are effectively copies of a repository where we can make (possibly experimental) changes without modifying the parent files. In most cases, the parent repository is the master branch, and we can create a new topic branch by using checkout with the -b flag:

$ git checkout -b modify-README
Switched to a new branch 'modify-README'
$ git branch
master
* modify-README

Here the second command, git branch, just lists all the local branches, and the asterisk * identifies which branch we’re currently on. Note that git checkout -b modify-README both creates a new branch and switches to it, as indicated by the asterisk in front of the modify-README branch. (If you set up the co alias in Sección 1.3, you can use git co -b modify-README instead.)

The full value of branching only becomes clear when working on a project with multiple developers,21 but branches are helpful even for a single-developer tutorial such as this one. In particular, the master branch is insulated from any changes we make to the topic branch, so even if we really screw things up we can always abandon the changes by checking out the master branch and deleting the topic branch. We’ll see how to do this at the end of the Sección.

By the way, for a change as small as this one I wouldn’t normally bother with a new branch, but it’s never too early to start practicing good habits.

Commit

With the changes made, we can take a look at the status of our branch:

$ git status
# On branch modify-README
# Changes to be committed:
#   (use "git reset HEAD <file>..." to unstage)
#
#       renamed:    README.rdoc -> README.md
#
# Changed but not updated:
#   (use "git add <file>..." to update what will be committed)
#   (use "git checkout -- <file>..." to discard changes in working directory)
#
#       modified:   README.md
#

At this point, we could use git add . as in Sección 1.3.2, but Git provides the -a flag as a shortcut for the (very common) case of committing all modifications to existing files (or files created using git mv, which don’t count as new files to Git):

$ git commit -a -m "Improve the README file"
2 files changed, 5 insertions(+), 243 deletions(-)
delete mode 100644 README.rdoc
create mode 100644 README.md

Be careful about using the -a flag improperly; if you have added any new files to the project since the last commit, you still have to tell Git about them using git add first.

Note that we write the commit message in the present tense. Git models commits as a series of patches, and in this context it makes sense to describe what each commit does, rather than what it did. Moreover, this usage matches up with the commit messages generated by Git commands themselves. See the GitHub post Shiny new commit styles for more information.

Merge

Now that we’ve finished making our changes, we’re ready to merge the results back into our master branch:

$ git checkout master
Switched to branch 'master'
$ git merge modify-README
Updating 34f06b7..2c92bef
Fast forward
README.rdoc     |  243 --------------------------------------------------
README.md       |    5 +
2 files changed, 5 insertions(+), 243 deletions(-)
delete mode 100644 README.rdoc
create mode 100644 README.md

Note that the Git output frequently includes things like 34f06b7, which are related to Git’s internal representation of repositories. Your exact results will differ in these details, but otherwise should essentially match the output shown above.

After you’ve merged in the changes, you can tidy up your branches by deleting the topic branch using git branch -d if you’re done with it:

$ git branch -d modify-README
Deleted branch modify-README (was 2c92bef).

This step is optional, and in fact it’s quite common to leave the topic branch intact. This way you can switch back and forth between the topic and master branches, merging in changes every time you reach a natural stopping point.

As mentioned above, it’s also possible to abandon your topic branch changes, in this case with git branch -D:

# For illustration only; don't do this unless you mess up a branch
$ git checkout -b topic-branch
$ <really screw up the branch>
$ git add .
$ git commit -a -m "Major screw up"
$ git checkout master
$ git branch -D topic-branch

Unlike the -d flag, the -D flag will delete the branch even though we haven’t merged in the changes.

Now that we’ve updated the README, we can push the changes up to GitHub to see the result. Since we have already done one push (Sección 1.3.4), on most systems we can omit origin master, and simply run git push:

$ git push

1.4 Deploying

Even at this early stage, we’re already going to deploy our (still-empty) Rails application to production. This step is optional, but deploying early and often allows us to catch any deployment problems early in our development cycle. The alternative—deploying only after laborious effort sealed away in a development environment—often leads to terrible integration headaches when launch time comes.22

Deploying Rails applications used to be a pain, but the Rails deployment ecosystem has matured rapidly in the past few years, and now there are several great options. These include shared hosts or virtual private servers running Phusion Passenger (a module for the Apache and Nginx23 web servers), full-service deployment companies such as Engine Yard and Rails Machine, and cloud deployment services such as Engine Yard Cloud and Heroku.

My favorite Rails deployment option is Heroku, which is a hosted platform built specifically for deploying Rails and other web applications.24 Heroku makes deploying Rails applications ridiculously easy—as long as your source code is under version control with Git. (This is yet another reason to follow the Git setup steps in Sección 1.3 if you haven’t already.) The rest of this Sección is dedicated to deploying our first application to Heroku.

1.4.1 Heroku setup

Heroku uses the PostgreSQL database (pronounced “post-gres-cue-ell”, and often called “Postgres” for short), which means that we need to add the pg gem in the production environment to allow Rails to talk to Postgres:

group :production do
  gem 'pg', '0.12.2'
end

Appending this code to the Gemfile from Listing 1.5 yields Listing 1.9.

Listing 1.9. A Gemfile with an added pg gem for PostgreSQL. 
source 'https://rubygems.org'

gem 'rails', '3.2.13'

group :development do
  gem 'sqlite3', '1.3.5'
end


# Gems used only for assets and not required
# in production environments by default.
group :assets do
  gem 'sass-rails',   '3.2.5'
  gem 'coffee-rails', '3.2.2'

  gem 'uglifier', '1.2.3'
end

gem 'jquery-rails', '2.0.2'

group :production do
  gem 'pg', '0.12.2'
end

To install it, we run bundle install with a special flag:

$ bundle install --without production

The --without production option prevents the local installation of any production gems, which in this case is just pg. (Because the only gem we’ve added is restricted to a production environment, right now this command doesn’t actually install any additional local gems, but it’s needed to update Gemfile.lock since that’s what Heroku uses to infer the gem requirements of our application.)

Next we have to create and configure a new Heroku account. The first step is to sign up for Heroku; after checking your email to complete the creation of your account, install the necessary Heroku software using the Heroku Toolbelt.25 Then use the heroku command to log in at the command line (you may have to exit and restart your terminal program first):

$ heroku login

Finally, navigate back to your Rails project directory and use the heroku command to create a place on the Heroku servers for the sample app to live (Listing 1.10).

Listing 1.10. Creating a new application at Heroku. 
$ cd ~/rails_projects/first_app
$ heroku create
Created http://stormy-cloud-5881.herokuapp.com/ |
git@heroku.com:stormy-cloud-5881.herokuapp.com
Git remote heroku added

The heroku command creates a new subdomain just for our application, available for immediate viewing. There’s nothing there yet, though, so let’s get busy deploying.

1.4.4 Heroku commands

There are many Heroku commands, and we’ll barely scratch the surface in this book. Let’s take a minute to show just one of them by renaming the application as follows:

$ heroku rename railstutorial

Don’t use this name yourself; it’s already taken by me! In fact, you probably shouldn’t bother with this step right now; using the default address supplied by Heroku is fine. But if you do want to rename your application, you can arrange for it to be reasonably secure by using a random or obscure subdomain, such as the following:

hwpcbmze.heroku.com
seyjhflo.heroku.com
jhyicevg.heroku.com

With a random subdomain like this, someone could visit your site only if you gave them the address. (By the way, as a preview of Ruby’s compact awesomeness, here’s the code I used to generate the random subdomains:

('a'..'z').to_a.shuffle[0..7].join

Pretty sweet.)

In addition to supporting subdomains, Heroku also supports custom domains. (In fact, the Ruby on Tutorial de Rails site lives at Heroku; if you’re reading this book online, you’re looking at a Heroku-hosted site right now!) See the Heroku documentation for more information about custom domains and other Heroku topics.
